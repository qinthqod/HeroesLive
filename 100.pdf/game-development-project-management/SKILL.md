# 游戏开发项目管理

## 触发场景
- 规划项目资源分配和优先级
- 制定开发里程碑和风险评估计划
- 面临进度、成本、质量的权衡决策
- 设计迭代开发流程

## 核心决策框架

### 一、三选二权衡法则

在快速、便宜、优质三个目标中，通常只能同时实现两个：

| 组合 | 结果 | 适用场景 |
|------|------|----------|
| 快速 + 优质 | 不便宜（昂贵） | 有野心的正式开发阶段 |
| 便宜 + 优质 | 不快速 | 独立开发者长期项目 |
| 快速 + 便宜 | 不优质 | 原型阶段、次要功能 |

**布鲁克斯定律警示**："9个女人加起来也并不能在一个月内生出一个孩子"——某些项目有不可压缩的时间下限。

**动态调整策略**：
- 初始阶段：强调快速和便宜（原型验证）
- 正式开发：强调快速和优质（接受高成本）
- 次要功能：牺牲质量换取速度

### 二、80/20资源分配法则

**核心原则**：80%的价值由20%的功能驱动。

**执行步骤**：
1. 识别核心功能：玩家80%时间使用的行为（移动、战斗、跳跃）
2. 集中资源：将80%的开发精力投入这20%的核心功能
3. 避免陷阱：不要在花哨的次要功能上过度投入

**案例参考**：
- 《塞尔达传说》：重点优化移动和战斗，而非蜡烛/炸弹等道具
- 《魔兽世界》：优先提升角色移动速度

### 三、风险评估时机决策

**步骤1：判断开发方法**
```
IF 瀑布式开发 → 在里程碑和截止期限进行风险评估
IF 敏捷开发 → 在Sprint规划会上进行风险评估
```

**步骤2：确定执行主体**
- 个人承担：可行，但视角有限
- 团队承担：推荐，有更强组织结构支持深入讨论

### 四、迭代开发流程

**定义**：将前一次迭代的输出作为下一次迭代的输入。

**标准流程**：
1. 起点：粗略图形（形状和字母）+ 基础功能的原型
2. 迭代增补：
   - 添加更多游戏功能
   - 现有机制变得更复杂强大
   - 图形和声音逐步完善
3. 目标：完善、反应灵敏、无障碍的成品

**游戏内迭代设计**：
- 塔防游戏：每波敌人是新迭代（敌人变强，炮塔进步）
- 《大金刚》：通关循环，难度提升
- 《银河战士》：能力成长，解锁新区域

## 决策流程图

```
项目规划
    ↓
应用三选二法则确定优先级
    ↓
用80/20法则分配资源
    ↓
根据开发方法确定风险评估时机
    ↓
执行迭代开发
    ↓
持续评估与调整
```

## 注意事项
- 反向规律不成立：项目可能既慢、又贵、且劣质
- 需平衡核心体验与次要功能的完整性
- 不适用于未建立明确开发流程的混乱状态
