# 游戏架构决策

## 何时使用
- 游戏开发前期的架构设计阶段
- 确定游戏整体结构
- 定义游戏核心主题
- 评估新功能与主题的一致性
- 关卡设计规划

## 沙盒式体验

### 特征判定
- 提供自由空间，无明确特定目标
- 核心是探索与实验
- 机制需简单以便创造性组合

### 设计要求

**机制设计**：
- 开发者优先处理显而易见的组合
- 预测边缘情况
- 确保组合逻辑通顺

**环境设计**：
- 若仿照现实，需处理写实细节
- 考虑人口、天气等因素
- 避免破坏沉浸感

### 注意事项
- 沙盒不等于开放世界
- 开放世界是沙盒的优秀呈现机制
- 沙盒可以是游戏的一小部分

## 导轨式体验

### 特征判定
- 相对线性的完成方式
- 集中玩家注意力
- 可包含多路径选择

### 节奏控制模型

**火车模型**：
- 长时间做相同的事
- 无曲折，保证顺利
- 适合叙事驱动游戏

**过山车模型**：
- 方向突变
- 视觉刺激
- 控制预期带来兴奋
- 适合动作游戏

## 决策依据
分析以下因素：
1. 游戏核心（参考原理41）
2. 目标受众（参考原理60）
3. 游戏主题（参考原理58）

## 组合策略

### 沙盒游戏
- 常以导轨式体验开始（教学/适应）
- 逐步开放自由度
- 引导玩家进入沙盒

### 导轨式游戏
- 可插入沙盒段落
- 提供自由探索区域
- 平衡线性与自由

### 注意事项
- 结合需精细考量
- 两者吸引不同类型玩家
- 避免体验割裂

## 主题定义

### 确立主题
**定义**：主题是设计传达的中心思想，驱动所有工作，赋予游戏娱乐之外的价值。

**要求**：
- 必须明确
- 必须有目标性
- 不可过于宽泛

**示例**：
- ❌ 单纯的“战争”
- ✅ “战争的可怕”（强调平民伤亡、创伤）
- ✅ “战士的荣耀”（强调英雄主义、爱国）

### 一致性检查

**决策逻辑**：
```
IF 添加新功能 THEN
    检查是否与主题冲突？
    IF 冲突 THEN
        拒绝或修改
        避免传达混乱信息
    ELSE IF 呼应或强化 THEN
        采纳
```

**冲突示例**：
- 恐怖场景配激昂音乐
- 战争主题加入滑稽元素
- 严肃剧情加入无厘头任务

### 主题与设定的关系

**独立性**：
- 主题独立于体裁和设定
- 同一主题可置于不同背景

**示例**：
- 主题“爱战胜一切”
- 可置于：亚瑟王宫殿、现代战场、外太空

### 叙事结合

**主角故事框架**：
- 利用主角的故事表达主题
- 主角学到的即希望玩家思考的
- 例如：主角学会无私 → 玩家思考无私的价值

## 决策流程

### 第一步：明确核心
- 确定游戏的核心玩法
- 定义核心循环

### 第二步：分析受众
- 了解目标玩家群体
- 分析玩家偏好

### 第三步：定义主题
- 确定中心思想
- 确保主题明确

### 第四步：选择结构
- 基于核心、受众、主题
- 选择沙盒、导轨或混合

### 第五步：持续验证
- 新功能与主题一致性检查
- 结构与核心匹配度检查
