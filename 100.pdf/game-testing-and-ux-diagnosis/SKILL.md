# 游戏测试与用户体验诊断

## 触发场景
- 规划游戏测试流程
- 执行用户测试并收集反馈
- 诊断玩家在特定环节受阻的原因
- 优化游戏体验和关卡设计

## 第一部分：测试方法分类

### 测试方法选择决策树

```
测试目标？
├── 验证首次体验 → 一次性测试
├── 发现系统漏洞 → 黑盒测试
├── 验证特定功能 → 白盒测试
└── 验证服务器承载 → 负载测试
```

### 1. 一次性测试

**执行方式**：受试者只能使用一次，用完即弃（像面巾纸）

**目的**：
- 验证玩家第一次看到游戏时的反应
- 检验概念和机制的理解程度
- 确认游戏是否传达了"正在发生什么"以及"如何继续"

**关键点**：一旦受试者学会了操作，其价值即丧失。

### 2. 黑盒测试

**执行方式**：受试者对游戏运作方式完全不知情，直接开始玩

**目的**：
- 发现系统漏洞
- 展示玩家实际游玩方式（常与开发者意图不同）

**时机**：
- 及早安排以排除假设
- 后期持续进行以符合玩家行为形态

### 3. 白盒测试

**执行方式**：受试者了解特定情形下应发生什么，根据测试脚本操作

**目的**：发现大部分漏洞和问题

**后续步骤**：问题解决后必须进行回归测试，确保修复未影响其他部分。

### 4. 负载测试

**执行方式**：
- 一定数量测试人员同时进入游戏
- 或自动模拟成千上万玩家

**目的**：确保代码和服务器能支撑大量数据包的处理

**适用**：在线/多人游戏

### 最终目标检查
无论哪种测试，最终都要确保游戏体验是有趣的。如果受试者觉得不好玩，玩家也不会觉得好玩。

## 第二部分：玩家问题解决障碍诊断

### 四类障碍诊断

当玩家在游戏中无法解决问题或做出决策时，按以下维度诊断：

### 1. 功能固着

**现象**：玩家理解了一种解法，但当问题经过小调整后就无法解决。

**示例**：
- 玩家知道打破棕色箱子获取物品
- 面对蓝色箱子时不知所措（是打破？射它？站上去？）

**对策**：引导玩家改变思维方式

### 2. 无关信息

**现象**：玩家无法区分哪些信息与手边问题相关，被过多细节干扰。

**示例**：
- 双手持手枪且无法换武器
- 仍去关注刀或火箭筒

**对策**：提供恰到好处的信息量，保持沉浸感的同时避免干扰

### 3. 假设

**现象**：玩家基于错误信息做出假设，导致停止游戏。

**示例**：认为目标太远而不起跳

**对策**：
- 提供存档点减少失败成本
- 通过视觉、声音线索提示安全性
- 引入新能力（如爬墙）

### 4. 心理定势

**现象**：玩家仅具备从以往经验学习的"常识"，难以切换到新的机制或视角。

**示例**：
- 在《魂斗罗》中习惯了2D横向卷轴操作
- 视角切换为3D第三人称射击时，尽管按键功能相同，仅美术改变就导致玩家跟不上

**对策**：提供渐进式教程或过渡关卡
