# 青岚夜行：游戏设计执行准则

本文件由项目内现有 Godot 规则、完整性检查、玩法说明、美术方向和 Web 版问题审计提炼而成。后续开发以此作为功能与内容验收依据。

## 1. 牌组不是卡牌列表，而是循环系统

- 战斗必须具有抽牌堆、手牌、弃牌堆、消耗区和洗牌。
- 打出的牌必须离开手牌；回合结束弃置剩余手牌并重新抽取。
- 抽牌动作必须能让玩家理解牌从哪里来、进入哪里以及牌堆是否洗回。
- 手牌上限默认 7，基础每回合抽 5。
- 心魔牌不可打出，会真实占据手牌并进入弃牌循环。
- 所有伤害来源必须汇入同一个胜利结算出口，包括直接伤害、燃烧、丹毒、机关和其他回合末效果；任何敌人生命归零都必须进入奖励或章节结算。

## 2. 每张牌必须提供一个决策理由

卡牌最少包含：

- 费用、类型、稀有度
- 一个清晰的主效果
- 一个关键词
- 一个条件联动或构筑用途
- 一个明确的精研变化

禁止仅替换名称、插图而复用完全相同的效果。每个职业至少拥有 8 类可辨识关键词和 3 条成型构筑路线。

- 卡牌必须使用稳定运行时标识，不能依赖展示名称或模糊文本推断全部效果。
- 真解数值必须依据精研说明生成；没有写明减费的牌不得自动降低费用。
- 手牌上限为 7 时，界面和快捷键必须同时支持 7 张，额外抽牌不能进入不可见区域。
- 每次扩充牌池都要通过运行时覆盖和职业预算检查。

## 3. 职业差异来自资源循环

- 剑修：剑势积蓄、连击、终结。
- 符师：符印附着、燃烧、集中引爆。
- 丹师：寒热药性、丹毒、治疗转换。
- 御灵：灵兽切换、指令协同。
- 偃师：机关部署、机巧升级、回收再触发。
- 契师：魂灯、生命交易、弃牌复用。

职业专属资源必须在卡牌文字、战斗状态和奖励推荐中同时可见。

新局顺序固定为：选择角色与职业 → 选择章节 → 进入章节剧情。角色选择不应藏在章节之后。

12 张起始牌组由 6–8 种基础牌按功能合理复数组成，不得直接塞入全部基础牌或提前赠送真解；局内战利必须保留真实的新牌发现空间。

职业熟练度必须形成可预期的固定里程碑，而不是只显示进度条。25/50/75/100 熟练分别改变开局经济、起始牌组、职业资源和初始法宝；职业页需同时展示已获得奖励与下一目标。

## 4. 奖励必须随章节与构筑变化

- 第一幕稀有度权重：普通 60 / 精良 30 / 稀有 10。
- 第二幕：普通 42 / 精良 38 / 稀有 20。
- 第三幕及以后：普通 34 / 精良 36 / 稀有 30。
- 第二幕开始奖励可直接出现精研牌，后期概率更高。
- 奖励要展示与当前牌组的契合点，而不是只展示卡面。
- 三选一至少有一张能够推进当前最接近且尚未完成的流派图谱；另外两张保留即时补强和转型可能。
- 奖励页应展示当前构筑方向、核心组件进度与本张牌的推进理由，避免把推荐等级变成不可解释的黑箱。

## 5. 路线选择必须回答三个问题

每个节点在点击前必须让玩家知道：

1. 风险是什么；
2. 主要收益是什么；
3. 选择会对本局或故事造成什么影响。

路线页只突出当前可选节点，已走路线作为轨迹，未来路线作为预告，避免将所有节点做成同等权重的暗色方框。

## 6. 叙事必须进入系统

- 重要选择写入本局印记或人物关系。
- 路线、奖励和战斗应回应已经发生的剧情。
- 每章至少包含开端、冲突、揭示和章末悬念。
- 叙事选择不能只有文字差异，至少影响一个资源、节点、敌人或后续对话。
- 选择按钮在点击前展示明确后果；已经发生的选择进入异闻录，并跨局保留为剧情印记。
- 每章必须定义一个可追踪调查目标、开场线索、四层路线证据与章末结论。不同路线提供不同证据，调息或修炼等安全选择允许放弃调查进度。
- 章末需区分“真相已明”和“疑云未尽”，并提示重返章节选择其他路线的价值。
- 线索只能在节点完成后结算：战斗需获胜、异闻需作出选择、坊市需离开。仅进入节点不得提前获得证据；失败复盘必须说明哪条待查证线索未能带回。
- 单局获得 4 条证据即可完成章节叙事；每章全部 7 条分支证据进入跨局调查宗卷。宗卷补证需跨局合并，全部收录后只发放一次悟道与灵玉奖励。
- 战利、法宝、跳过奖励与宗卷完成奖励共享单次结算锁，快速连点不得重复获得卡牌、货币或成长资源。
- 五章奇遇不得复用同一组选项。每章提供构筑成长、生存恢复、高收益代价和谨慎离开四类策略；所有数值代价在点击前明示。
- 奇遇只有实际调查相关内容后才结算路线证据；无收益离开会保住当前状态，但放弃该分支线索。
- 五章坊市必须具有独立名称、货架倾向、价格结构和一次性专属交易；路线入口名称与实际页面必须一致。
- 坊市商品需展示对当前构筑的契合理由。低费、净心、高费真解、循环稀有和终局传承应随章节形成供给升级，避免所有市场成为同一随机卡池。

## 7. 法宝是局内被动层

- 法宝来自精英奖励、坊市或事件，与拿牌形成明确取舍。
- 每件法宝必须拥有可执行字段，至少影响开局、出牌、状态结算、战后恢复、经济或牌组资源之一。
- 法宝图标、名称和当前效果应在战斗与路线中可查看，并写入自动存档。
- 获得过的法宝进入异闻录；数据检查必须覆盖唯一 ID、唯一名称、素材存在和可执行效果。

## 8. 移动端信息优先级

- 同屏优先级：敌人意图 > 手牌可读性 > 玩家资源 > 场景装饰。
- 敌人不能只重复单一攻击；普通、精英和首领至少具备攻击、蓄势/护体和高压招式组成的可预判循环。
- 普通战负责教授本章单一压力，精英战负责组合考核；十场普通与精英战不得共享同一套换名循环。
- 每个敌人必须公开“战斗原型、核心特性、反制建议”，但反制提示只指出决策方向，不替玩家指定唯一出牌顺序。
- 章节压力按“蓄势与连击 → 灵气压制 → 虚弱与多段雷击 → 抽牌削减与恢复 → 牌堆污染与复合封锁”递进，使前章机制成为后章的学习基础。
- 五章首领必须各自拥有“序—破—急”三式循环，至少有一种章节专属压力：灵气压制、多段雷击、抽牌削减、恢复或心魔入牌堆。
- 当前招式与下一式同时可见，数值必须明确区分单次伤害、多段伤害、护体和状态施加。
- 战斗栏中的消耗品必须可以真实使用、扣除数量并进入统一伤害或资源结算。
- 卡牌主效果必须在 430×932 下可辨识；联动说明允许更小，但不能被裁切。
- 琥珀色只用于推荐、当前路径和主要动作；玉青用于导航、防御与工具；朱砂用于危险。
- 实景插画承担氛围，UI 表面承担信息，不以大面积半透明黑框替代层级设计。

首次战斗采用三步渐进式引导：先读当前与下一招式，再单击卡牌立即施放，最后结束回合。每次只解释一个动作，不暂停战斗、不接管玩家输入，并允许跳过；完成状态跨局保存。

出牌、命中、护体、治疗、抽牌、敌方行动、受击和获得奖励需要提供节制的即时声音反馈；支持设备可同步触觉反馈。音效、触觉与音量必须允许独立控制并持久化，关闭反馈不得影响玩法。

章末评阶必须来自可解释的真实表现，包括余命、战斗完成度、总回合和承伤；结算页展示的修为、悟道、灵玉、出牌与伤害必须与本局实际记录一致，禁止固定假数字。

藏经阁默认只收录各职业起始牌；战利、坊市、事件、精研和职业传承得到的卡牌永久入卷。未知卡牌只展示稀有度、类型、关键词或获取方向，不提前泄露完整名称与规则。

主城任务不得使用固定假进度。剧情场景阅读、人物手札和一次性奖励必须形成可恢复的跨局状态；奖励领取应具备幂等性，重复阅读不能重复发放。

## 9. 验收要求

- 数据检查：职业数量、卡牌数量、唯一名称、唯一效果、关键词覆盖。
- 流程检查：抽牌、出牌、弃牌、洗牌、敌方回合、奖励和下一战。
- 响应式检查：430×932、599×772、960×720。
- 内容检查：路线风险/收益/后果完整，奖励稀有度随章节变化。
- 视觉检查：路线和卡牌页面必须与现有水墨、旧金、玉青、夜蓝体系一致。

Web 数据与内容门禁：

```bash
cd web
npm run check:game
```

该检查会验证职业、卡牌、构筑、章节、路线、剧情数量，并检查每个职业的卡牌唯一性、效果差异、关键词覆盖、起始牌组和必要字段。
## 构筑图谱与长期收集

- “100 种以上卡组组合”必须由真实唯一的卡牌集合支撑，不以重复数据改名充数。
- 每套图谱固定展示五张核心组件，并按入门、进阶、真传形成能力梯度。
- 未收录卡牌保留未知感，但展示缺失数量与关键词线索，兼顾探索感和可行动性。
- 完成图谱后揭示完整出牌思路，让收集奖励直接服务于下一局构筑，而不是只增加数字。
- 局内当前流派目标必须贯穿路线、战斗和牌组卷册：路线说明方向，战斗只保留名称与进度，卷册承载缺失核心与完整打法。
- 真解术法视为同名基础术法的上位形态，精研不得导致局内流派完成度倒退。
